# 项目全景 · autohttps

> 文档状态: draft · 模块: project · 端点: app · 撰写: product-manager
> 信任基础: 已批准的需求 brainstorming 决策(2026-07)+ 2026-07-16 关键决策确认(端点形态 / MVP 范围 / 证书交付边界 / 访问鉴权)

---

## 1. 项目定位

autohttps 是一个跨 Windows / Linux / macOS 的 HTTPS 证书全生命周期管理工具,提供两种运行形态:

- **桌面形态**:面向个人开发者的 Tauri 桌面应用,主窗口 + 系统托盘常驻;
- **服务器形态**:面向运维 / 团队的 7×24 守护进程,通过浏览器访问 Web UI。

两种形态共享同一份 Rust 核心代码与同一份 React + Vite 前端,产品定位都是 **让"维护一张域名列表 + 一组证书"这件事变得省心**。

---

## 2. 目标用户

| 角色       | 一句话需求                                          | 典型使用场景                              |
| ---------- | --------------------------------------------------- | ----------------------------------------- |
| 个人开发者 | 本地起几个内网 / 公网域名,证书别让我半夜起来续      | 桌面形态,启动后看到列表红点就知道谁该续   |
| 小团队运维 | 几十个域名分散在多台机器,证书过期前必须有人看到提醒 | 服务器形态,跑在网关机器,Web UI 给团队共用 |
| 平台 / SRE | 需要 API + CLI 自动化集成到自己的运维体系           | 服务器形态 + HTTP API 集成(MVP 后置)     |

---

## 3. 产品价值主张

| 痛点                                    | autohttps 的应对                               |
| --------------------------------------- | ---------------------------------------------- |
| 证书突然过期,服务挂掉                   | 启动即检测;到期前按策略自动续签;失败有清晰错误 |
| ACME 配置零碎,各家 DNS / webroot 各不同 | 把 HTTP-01(webroot)与 DNS-01(手动)封装成向导   |
| 多台机器 / 多域名状态零散,集中看不到    | 一张域名列表 + 状态总览 + 任务历史             |
| 内网开发需要自签证书,根 CA 管理混乱     | 提供自签根 CA 模块,统一签发                    |

---

## 4. 运行形态与端点

工作流层面**只有一个端点 `app`**:桌面形态与服务器形态**共享同一份前端**,页面契约与原型只维护一套。两种形态的差异通过**运行模式**在运行时区分,而非拆分成两个端。

| 运行模式     | 载体             | 交互入口              | 形态特有能力                          |
| ------------ | ---------------- | --------------------- | ------------------------------------- |
| 桌面模式     | Tauri 桌面应用   | 主窗口 + 系统托盘     | 托盘常驻、关窗不退出、开机自启(可选)   |
| 服务器模式   | 守护进程         | 浏览器访问 Web UI     | 7×24 常驻、异常自恢复、监听地址/端口   |

- 两种形态是**各自独立的部署实例**,各自持有各自的数据,形态间不共享域名/证书/任务数据。
- 页面 PRD 路径统一为 `pages/app/{模块}/{页面}.md`;个别页面/控件的形态差异在页面 PRD 内以"仅桌面 / 仅服务器"标注,不另开端点。

## 5. 模块划分(MVP)

MVP 拆分为 **7 个模块**。下游 flow / 模块 PRD / 页面 PRD 均以此为准。

| #   | 模块       | slug           | 职责(业务口径)                                                          |
| --- | ---------- | -------------- | ----------------------------------------------------------------------- |
| 1   | 总览仪表盘 | `dashboard`    | 状态总览(证书总数 / 即将到期 / 失败数);红点与待处理高亮;快速入口       |
| 2   | 域名管理   | `domains`      | 域名列表维护(增删改查);域名与证书、验证方式的关联;备注 / 分组          |
| 3   | 证书管理   | `certificates` | 证书列表与详情;手动签发 / 续签触发;吊销 / 删除;证书文件导出            |
| 4   | ACME 签发  | `acme`         | ACME 账户配置;验证方式向导:HTTP-01(webroot)与 DNS-01(手动);挑战执行 |
| 5   | 自签根 CA  | `local-ca`     | 创建 / 导入自签根 CA;用根 CA 签发内网证书;导出根 CA 供客户端信任        |
| 6   | 任务与历史 | `tasks`        | 签发 / 续签任务的队列与执行历史;每次执行的时间 / 结果 / 日志 / 重试      |
| 7   | 系统设置   | `settings`     | 运行形态配置;存储路径;全局续签策略(提前天数 / 自动开关);ACME 默认账户 |

**MVP 内优先级**:P0 证书主线(`dashboard` / `domains` / `certificates` / `acme` / `tasks`);P1 `local-ca` / `settings`。

**通知**:MVP 阶段退化为总览页的**红点 / 到期高亮**(归 `dashboard`),不单列独立模块。

## 6. 产品范围与非目标

### 6.1 MVP 做什么

- 维护域名列表,围绕域名管理其证书的完整生命周期(签发 → 续签 → 到期 / 失败可见)。
- ACME 自动签发:支持 HTTP-01(webroot)与 DNS-01(手动添加 TXT 记录)两种验证方式。
- 自签根 CA:内网 / 本地开发场景下统一签发可信证书。
- 证书交付止于**签发 + 本地存储 + 到期/失败提醒**;证书文件可导出,由用户自行部署到目标服务。
- 跨平台桌面与服务器两形态,共享前端。

### 6.2 MVP 明确不做(非目标)

- ❌ **证书自动部署 / 安装**到 nginx / apache / 系统证书库,或执行部署脚本(deploy hook)。
- ❌ **通知多渠道**(邮件 / Webhook / IM):MVP 仅 UI 红点提醒。
- ❌ **HTTP API / CLI 集成**:面向 SRE 的自动化接口后置。
- ❌ **Web UI 鉴权与多用户权限**:MVP 不做登录,默认仅本机 / 可信内网访问。
- ❌ **DNS-01 厂商 API 自动验证**(Cloudflare / 阿里云等):MVP 仅手动 DNS-01。

> 非目标均为"后续可扩展",不是"永不做";理由见 §9 决策记录。

## 7. 关键约束与非功能需求

| 维度       | 约束(业务口径)                                                                 |
| ---------- | -------------------------------------------------------------------------------- |
| 跨平台     | Windows / Linux / macOS 三平台功能对等                                            |
| 常驻       | 桌面形态托盘常驻、关窗不退出;服务器形态 7×24 守护、异常自恢复                     |
| 启动即检测 | 两形态启动后立即扫描全部证书状态,过期 / 即将过期项第一时间可见                    |
| 数据安全   | 证书私钥、ACME 账户密钥属敏感数据,本地安全存储、不明文外泄(具体机制由 architect 定) |
| 离线可用   | 自签 CA 与已签发证书的查看 / 管理需在无外网环境可用;ACME 在线签发依赖外网          |
| 访问边界   | 服务器形态 Web UI 默认仅监听本机 / 可信内网;暴露公网需用户自行加反向代理鉴权,产品内明确提示风险 |
| 数据持久化 | 域名 / 证书 / 任务记录本地持久化;两形态实例各自独立,互不共享                      |

MVP 一次交付上述 7 个模块(§5),内部按 P0 → P1 推进。后置 backlog(§6.2 非目标)按需排期,不纳入当前流水线。

## 9. 决策记录(append-only)

> 只增不改;每条记"决定了什么 / 为什么不做另一选项"。

- **D1(2026-07-16)· 端点单端**:桌面与服务器统一为一个 `app` 端,不拆 `desktop` / `server`。
  - 为什么不拆:两形态共享同一前端,拆端会让页面契约与原型重复维护、成本翻倍;形态差异用运行模式 + 页面内标注即可覆盖。
- **D2(2026-07-16)· 标准 MVP 范围**:交付 7 模块(§5);通知多渠道、HTTP API / CLI 后置。
  - 为什么不做全量:先把"域名 + 证书 + 自动续签 + 可见提醒"这条核心主线跑通;通知渠道与自动化接口价值真实但非首版必需。
- **D3(2026-07-16)· 证书交付边界**:止于签发 + 存储 + 提醒,不做自动部署。
  - 为什么不做部署:目标服务(nginx / apache / 各类中间件)与部署方式差异极大,自动部署复杂度与风险显著高于收益;MVP 以"导出 + 用户自行部署"替代。
- **D4(2026-07-16)· MVP 不做鉴权**:服务器 Web UI 不做登录 / 多用户权限,默认仅本机 / 可信内网访问。
  - 为什么不做:小团队共用一台网关机的信任内网是主场景;多用户权限矩阵成本高;公网暴露以"自行加反代鉴权 + 产品内风险提示"过渡。
