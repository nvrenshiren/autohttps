## 1. 概述

acme 是证书主线中"**证明域名控制权**"的落点模块。它对外提供两件事:**ACME 账户配置**(在 CA 上建立可用于申请证书的身份),与**验证方式向导 + 挑战执行**(为域名配置 HTTP-01 或 DNS-01 验证方式,并在被委托时向 CA 完成校验)。

本模块**聚焦验证与取证过程**:被 `certificates` 委托对某组域名执行验证,验证通过后完成 ACME 订单**取得证书并交回** certificates;certificates 负责证书的存储、状态与生命周期(引用不复述)。验证方式的**关联**(某域名用哪种方式)归 `domains`;每次执行经 `tasks` 留痕;默认账户的指定归 `settings`;"等待手动添加 TXT"等待处理提示由 `dashboard` 呈现。

遵 project §6.2:MVP **仅手动 DNS-01**,不做 DNS 厂商 API 自动验证。MVP 单一角色 operator 对本模块全权(roles.md),无细粒度权限差异。

## 2. 功能列表(按端分组)

> 端点只有一个 `app`(桌面形态 / 服务器形态共享前端);形态差异在功能项内以"仅桌面 / 仅服务器"标注。

### 2.1 端 `app`

**A. ACME 账户管理**

| 功能 | 说明 |
| --- | --- |
| A1 配置并注册 ACME 账户 | 选择目标 CA、填写联系邮箱、同意服务条款,发起注册;推进账户状态机至"已注册" |
| A2 查看 ACME 账户 | 账户列表与状态(未配置 / 注册中 / 已注册 / 注册失败)、目标 CA、CA 环境(生产 / 测试)、联系邮箱;标示哪个为默认(默认的设定归 settings,此处只读呈现) |
| A3 编辑 ACME 账户 | 更新联系邮箱等;必要时对 CA 更新账户信息,业务上仍为同一账户 |
| A4 注册失败重试 | 对"注册失败"账户修正后重试注册 |
| A5 移除 ACME 账户 | 移除本地账户配置,退出账户状态机;若有证书正引用该账户签发,需提示影响并二次确认 |

**B. 验证方式向导与挑战执行**

| 功能 | 说明 |
| --- | --- |
| B1 验证方式选择向导 | 为域名选择验证方式:HTTP-01(webroot)或 DNS-01(手动);通配符域名强制 DNS-01(glossary);把"该域名用哪种方式"的关联落 domains,acme 保留执行参数 |
| B2 HTTP-01 配置 | 指定该域名 HTTP-01 验证所需的 webroot 路径(验证文件放置位置) |
| B3 执行验证挑战 | 被 certificates 委托时,针对每个域名建立订单、取得挑战、推进挑战状态机(见 flows/acme.md);对首签 / 续签不作区分 |
| B4 DNS-01 手动引导 | 展示待添加的 TXT 记录(名 + 值,可复制)、确认已添加、触发校验;可选提交前本地预检 TXT 是否生效;承接自动续签遗留的"等待手动配置"挑战 |
| B5 挑战失败重试 | 对"验证失败"挑战重试(HTTP-01 重放验证文件 / DNS-01 修正 TXT 后重新校验);失败原因日志见 tasks |
| B6 查看验证 / 挑战状态 | 查看某次签发 / 续签下各域名挑战的当前状态、待处理项与失败原因概要 |

> 本模块**不含**:DNS-01 厂商 API 自动验证(project §6.2)、证书存储 / 到期提醒 / 自动续签调度(归 certificates / settings / tasks)、证书导出 / 吊销的对外动作(归 certificates)、批量账户操作与统计卡片。

## 4. 数据来源

> 供 architect 作为唯一设计依据,用**业务语言**描述本模块涉及哪些数据及其来源;**不含**表结构 / 字段类型 / 存储机制 / API(由 architect 设计)。敏感数据(账户密钥)的具体存储机制由 architect 依 project §7 数据安全约束确定。

| # | 数据 | 来源 | 说明 |
| --- | --- | --- | --- |
| DS1 | ACME 账户(本模块核心数据) | **本模块产生并维护** | 每个账户:目标 CA、CA 环境(生产 / 测试)、联系邮箱、服务条款同意情况、账户状态(账户状态机)、注册时间、账户密钥的存储位置引用(账户密钥属敏感数据,project §7) |
| DS2 | 验证方式执行配置 | **本模块产生并维护** | 域名采用 HTTP-01 时的 webroot 路径(验证文件放置位置);DNS-01 无常驻参数(挑战时动态生成 TXT)。域名↔验证方式的**关联**归 domains(见 §5-DEA5,已裁决) |
| DS3 | 验证挑战记录 | **本模块产生** | 每次被委托验证时,针对每个域名的挑战:验证方式、挑战状态(挑战状态机)、CA 下发的挑战要求(HTTP-01 验证文件内容 / DNS-01 待添加 TXT 的记录名与值)、结果、失败原因概要 |
| DS4 | 待验证域名 | `domains` | 验证对象是 domains 中已存在的域名;可含通配符(仅 DNS-01);域名的增删改归 domains |
| DS5 | 委托上下文(验证结果 / 取得的证书) | `certificates` | acme 被 certificates 委托执行验证并取证;只接收"验证哪些域名 / 用哪个账户签",把验证结果与**取得的证书**(叶子证书 + 证书链)交回;证书本体的存储、状态与生命周期归 certificates,acme 不留存 |
| DS6 | 默认 ACME 账户 | `settings` | 签发时若未显式指定账户,采用 settings 指定的默认账户;"哪个是默认"的设定归 settings,acme 消费不定义 |
| DS7 | 挑战执行任务与历史 | `tasks` | 挑战执行是签发 / 续签任务的一环;每次执行的时间 / 结果 / 日志 / 重试记录归 tasks;acme 推进挑战状态并把结果反映到任务,不复述任务历史 |

## 5. 决策记录(append-only)

> 只增不改;记"决定了什么 / 为什么不做另一选项"。与 flows/acme.md §5 互补(此处记模块 / 功能 / 数据相关决策)。

- **DEA1(2026-07-16)· 页面收敛为 2 页**:ACME 账户页 + 验证方式向导页(含挑战执行);不为 DNS-01 手动、HTTP-01 进度单列页面。
  - 为什么不拆:验证配置与其挑战执行在 DNS-01 首签本就连续,单列会割裂"配置方式"与"执行验证";自动续签遗留的手动挑战复用向导页的执行部分即可承接。若 CP2 设计发现单页过密,可再拆(属页面层决策)。
- **DEA2(2026-07-16)· 不做 DNS 厂商 API 自动验证**:承接 project §6.2 / §9;DNS-01 仅手动添加 TXT 记录。
  - 为什么不做:各家 DNS 厂商 API 差异大、凭据管理复杂;MVP 以"手动添加 TXT + 等待态提醒"覆盖,后续可扩展厂商插件。
- **DEA3(2026-07-16)· 支持配置多个 ACME 账户,默认账户归 settings**:acme 管理账户实体(可多),settings 指定默认。
  - 为什么:不同 CA、生产 / 测试环境需要多账户;与 certificates DS3"经哪个 ACME 账户"、settings"ACME 默认账户"口径一致。acme 不做"默认"逻辑,仅消费。
- **DEA4(2026-07-16)· acme 取证后交回,不留存证书本体**:acme 完成 ACME 订单取得证书即交回 certificates,自身不存储证书文件 / 私钥。
  - 为什么:证书的存储、状态、到期提醒、导出、吊销均归 certificates(其 DS6);acme 留存会造成双写与状态漂移。acme 仅持有账户密钥与挑战过程数据。
- **DEA5(2026-07-16)· 验证方式的执行配置归 acme,域名↔方式的关联归 domains**:acme 提供向导与挑战执行、持有执行参数(HTTP-01 webroot 路径);"某域名用哪种验证方式"的关联落 domains(project §5)。
  - 为什么:project §5 将"域名与验证方式的关联"划给 domains、"验证方式向导"划给 acme,据此切分。**已裁决(orchestrator, 2026-07-16):webroot 路径作为验证执行配置归 acme;domains 仅持"域名 ↔ 验证方式类别"关联。acme ↔ domains 边界对齐完成。**
