## 1. 概述

总览仪表盘是 autohttps 的**启动首屏**:使用者启动桌面 / 服务器形态后第一眼看到的页面。它把散落在各模块的证书状态**聚合成一屏可读的总览**——多少证书、谁即将到期、谁出了问题——并以**红点 / 待处理高亮**把「该处理的证书」推到眼前(project §1「启动即检测、看到红点就知道谁该续」)。

本模块是**纯聚合视图**,不产生自己的核心实体:证书状态与有效期来自 `certificates`,任务执行概况来自 `tasks`,待处理证书的域名经 `certificates` 携带自 `domains`。所有聚合**只读**,不判定、不改变任何证书状态(证书状态机是唯一真相,见 flows/certificates.md)。

本模块是 MVP 通知能力的落点:project §5 备注 / §6.2 明确 MVP 通知**退化为本模块的红点 / 到期高亮**,不做多渠道通知、不单列通知模块。

MVP 单一角色 operator 对本模块全权(roles.md「查看总览 / 红点 / 待处理项」),无细粒度权限差异。

## 2. 功能列表(按端分组)

> 端点只有一个 `app`(桌面形态 / 服务器形态共享前端);形态差异在功能项内以「仅桌面 / 仅服务器」标注。聚合口径与触发规则见 flows/dashboard.md,此处只列功能、不复述规则。

### 2.1 端 `app`

**A. 状态总览**

| 功能 | 说明 |
| --- | --- |
| A1 概览指标展示 | 展示三项概览指标:证书总数 / 即将到期数 / 失败数;口径见 flows/dashboard.md §2(即将到期数 + 失败数 = 全部待处理证书) |

**B. 待处理与红点提醒**

| 功能 | 说明 |
| --- | --- |
| B1 待处理清单 | 汇聚处于「即将到期 / 签发失败 / 续签失败 / 已过期」状态的证书,逐条呈现(含关联域名、当前状态、有效期);告警级(失败 / 已过期)优先于关注级(即将到期)。触发状态与级别见 flows/dashboard.md §3.1 |
| B2 红点提醒 | 存在任一待处理证书即亮红点,可显示待处理计数;全部处置完成后红点自动清零。桌面形态红点可同时体现在系统托盘图标角标(仅桌面);服务器形态体现在浏览器内(无托盘) |

**C. 快速入口**

| 功能 | 说明 |
| --- | --- |
| C1 待处理项跳转 | 从待处理项一键跳到该证书详情去处理(续签 / 重试 / 吊销 / 删除),或跳到其最近任务查看失败原因;去向见 flows/dashboard.md §4 |
| C2 常用操作入口 | 提供跳向「发起签发 / 查看全部证书 / 域名管理 / 任务与历史」的入口,便于从首屏直达常用操作 |

> 本模块**不含**:证书 / 任务 / 域名的任何增删改执行(仅跳转到对口模块执行)、证书状态的判定与扫描(归 certificates)、多渠道通知(project §6.2)、project §5 未明示的统计维度(如按签发方式 / CA 的分布图表)、多用户 / 审计视图(project §9-D4)。

## 4. 数据来源

> 供 architect 作为唯一设计依据,用**业务语言**描述本模块涉及哪些数据及其来源;**不含**表结构 / 字段类型 / 存储机制 / API(由 architect 设计)。本模块为纯聚合视图,以下数据**全部只读消费,不新增持久实体**。

| # | 数据 | 来源 | 说明 |
| --- | --- | --- | --- |
| DS1 | 证书当前状态与有效期(核心聚合源) | `certificates` | 每张证书的当前状态(证书状态机)、有效期(生效 / 失效时间),据此聚合三项指标并生成待处理清单;只读,不修改任何证书状态。「即将到期」等状态已由 certificates 依 settings 策略判定写入,本模块直接读取,不自行计算提前天数 |
| DS2 | 待处理证书的关联域名 | 经 `certificates` 携带(源自 `domains`) | 待处理项需显示「是哪个域名的证书该处理」;域名随证书数据一并取得,本模块不直接查询 domains |
| DS3 | 关联任务的执行结果与时间 | `tasks` | 待处理项(尤其失败类)关联其最近一次签发 / 续签任务的结果与时间,供「去排查失败原因 / 去处理」跳转;任务明细、日志、重试历史归 tasks,本模块只读展示概况、不复述 |
| DS4 | 运行形态 | 运行时环境 | 决定红点载体的呈现差异(桌面 = 系统托盘角标 + 窗口内 / 服务器 = 浏览器内);属呈现依据,非本模块持久业务数据 |

> **聚合基数(待 architect 关注)**:概览指标与待处理清单需覆盖**全部未删除证书**;三项指标口径(§2 / flows/dashboard.md §2)与红点触发集(flows/dashboard.md §3.1)须保持一致,避免出现「总览数字与待处理清单条数对不上」。刷新机制(实时 / 轮询 / 事件触发)由 architect 设计,业务要求仅为「呈现与证书当前状态一致」。

## 5. 决策记录(append-only)

> 只增不改;记「决定了什么 / 为什么不做另一选项」。与 flows/dashboard.md §6 互补(此处记模块 / 功能 / 数据 / 页面相关决策)。

- **DD1(2026-07-16)· dashboard 为纯聚合视图,不产生核心实体**:证书状态来自 certificates、任务概况来自 tasks、域名经 certificates 携带;本模块只读聚合呈现。
  - 为什么不自持数据:project §5 定位其为「总览 + 红点 + 快速入口」;自建证书 / 任务副本会与源模块产生双真相与漂移,违反单一出现原则。
- **DD2(2026-07-16)· 页面收敛为「总览页」1 页**:概览三指标、待处理清单与红点、快速入口均为同一首屏内的区块,不拆多页。
  - 为什么不拆:三者是「一眼看全 + 一键去办」的整体启动体验,拆页会割裂首屏叙事、增加跳转成本;需求(project §1/§5)也指向单一首屏。
- **DD3(2026-07-16)· 不做批量操作与额外统计维度**:待处理项仅提供逐条跳转到对口模块处理,不在 dashboard 内做批量续签 / 批量吊销;概览只保留 project §5 明示的三指标,不加按签发方式 / CA 的分布图表等。
  - 为什么:需求未明示批量与多维统计场景;证书的执行动作归 certificates、明细统计非首屏核心诉求,避免主动扩张功能(对齐 certificates DEC2 精神)。
- **DD4(2026-07-16)· 红点 / 高亮为 MVP 唯一通知形态**:承接 project §5 备注 / §6.2;通知能力在本模块以红点 + 待处理高亮落地,不做邮件 / Webhook / IM。
  - 为什么不做多渠道:MVP 主场景为本机 / 可信内网,启动即见的红点已满足「谁该续」核心诉求;多渠道通知后置(project §6.2),引入时按增量更新并同步 project 决策记录。
