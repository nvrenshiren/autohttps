# 业务流程与配置项 · 系统设置(settings)

> 文档状态: draft · 模块: settings · 端点: app · 撰写: product-manager
> 信任基础: project.md(approved)§4/§5/§6/§7/§9 · roles.md(operator 全权 / D4 无鉴权)· glossary.md(术语引用不复述)· flows/certificates.md(证书状态机 / 续签判定,**引用不复述**)
> **单一出现原则**:settings 是**配置项集合,无独立实体状态机**;续签相关状态(即将到期 / 续签中 / 续签失败等)定义在 flows/certificates.md,本文件仅说明"续签策略配置项如何被 certificates 消费",不复述其状态与流转。

---

## 1. 模块职责与边界

系统设置维护 autohttps 的**全局配置项集合**。这些配置不是围绕某个业务实体的生命周期,而是"一次设定、长期生效、被各模块读取"的全局旋钮。

### 1.1 本模块负责

- 定义并持久化四类全局配置:**运行形态配置**、**数据存储路径**、**全局续签策略**、**默认 ACME 账户**。
- 呈现当前配置、接受使用者修改并保存;根据当前运行形态(桌面 / 服务器)决定哪些配置项可见、可改。
- 作为**续签策略的唯一定义点**:certificates 消费本模块的续签提前天数与自动续签开关,本模块不判定证书状态、不执行续签。

### 1.2 委托与消费(边界)

| 事项 | 归属模块 | 本模块的关系 |
| --- | --- | --- |
| 续签提前天数、自动续签开关 | **本模块定义** | certificates **消费**本策略判定"即将到期"与是否自动续签;本模块只提供配置值,不定义证书状态机 |
| ACME 账户本体(创建 / 配置 / 邮箱 / 绑定 CA) | `acme` | 本模块仅从 acme 已配置的账户中**选择一个作为默认**;账户明细归 acme,本模块只存"默认指向哪个账户" |
| 证书 / 域名 / 任务 / 私钥 / 根 CA 的本地持久化 | 各自模块 | 本模块只定义这些数据落地的**存储根路径**;各数据的产生与结构归各自模块 |
| 当前运行形态(桌面 / 服务器) | 运行载体(部署既定) | 形态由部署载体决定(桌面 Tauri 应用 / 服务器守护进程),本模块**读取**该事实以决定形态相关项的可见性,**不提供"切换形态"操作**(project §4 / §9-D1 两形态各自独立部署) |

### 1.3 明确不做(遵 project §6.2 / §9-D4 / roles)

- ❌ 账户 / 登录 / 权限 / 会话等鉴权相关设置项(MVP 不做鉴权,roles §4)。
- ❌ 通知渠道配置(邮件 / Webhook / IM);MVP 提醒仅 dashboard 红点(project §6.2)。
- ❌ 主题 / 语言 / 外观 / 日志级别 / 更新检查等业务未明示的配置项。
- ❌ 独立的"续签重试次数 / 重试间隔"配置(见 §5-SF2)、"扫描 / 检测周期"配置(见 §5-SF3)。
- ❌ 运行形态之间的切换 / 数据迁移(两形态各自独立实例,project §4)。

## 2. 无独立状态机说明

settings **不具有实体生命周期状态机**。它管理的是一组**全局配置值**:每一项有"当前取值",使用者修改后保存即覆盖旧值并对后续行为生效,不存在"待生效 / 已作废"之类的状态流转。因此本文件以**配置项清单 + 影响范围 + 形态差异**取代状态机,不硬造状态图(见 §5-SF1)。

> 配置生效口径:配置保存后对后续读取该配置的模块行为生效。涉及运行载体的项(监听地址 / 端口、开机自启)其**生效时机**(即时 / 需重启守护进程 / 需重登录桌面会话)由 architect 依实现确定,本层不作机制承诺。

## 3. 配置项清单与业务含义

> 每项给出:业务含义、取值口径(默认由 architect 定合理值,本处只给业务约束)、所属形态。谁消费见 §4;形态汇总见 §5-表。

### 3.1 运行形态配置

| 配置项 | 业务含义 | 取值口径 | 形态 |
| --- | --- | --- | --- |
| 当前运行形态(只读) | 展示本实例是桌面形态还是服务器形态,让使用者明确当前能力集与可见设置项 | 只读,取自运行载体既定事实;不可在设置内切换 | 共有(取值不同) |
| 开机自启 | 桌面形态是否随操作系统登录自动启动并常驻托盘 | 开 / 关;对应 project §4"开机自启(可选)" | 仅桌面 |
| 监听地址 | 服务器形态守护进程对外提供 Web UI 的网络监听地址,决定谁能访问 | 默认仅监听本机 / 可信内网(project §7);若设为对外可达地址,等同把全部能力开放给任何可访问者,须在界面**明确提示公网暴露风险**(roles §3) | 仅服务器 |
| 监听端口 | 服务器形态 Web UI 的监听端口 | 由使用者指定,有合理默认 | 仅服务器 |

### 3.2 数据存储路径

| 配置项 | 业务含义 | 取值口径 | 形态 |
| --- | --- | --- | --- |
| 数据存储路径 | 所有本地持久化数据(域名 / 证书文件 / 私钥 / ACME 账户密钥 / 根 CA 材料 / 任务历史)落地的**根位置** | 一个本地目录,有合理默认;私钥与账户密钥属敏感数据,其安全存储机制由 architect 依 project §7 定;**路径在部署 / 首次启动时确定,运行期只读展示、不可更改,故无迁移语义**(见 §5-SF5) | 共有 |

### 3.3 全局续签策略(certificates 的唯一策略来源)

| 配置项 | 业务含义 | 取值口径 | 形态 |
| --- | --- | --- | --- |
| 续签提前天数 | 距证书到期还剩多少天时,判定该证书为"即将到期"——这是自动续签与红点提醒的触发阈值 | 正整数天数,有合理默认;certificates 以此判定"即将到期"(引用 certificates flow,不复述状态流转) | 共有 |
| 自动续签开关 | 是否在证书"即将到期"时自动发起续签;关闭则仅提醒不自动续,靠使用者手动续签 | 开 / 关;开启时,certificates 亦会在后续周期扫描中对"续签失败但仍未过期"的证书自动再尝试(即 certificates flow 所述"自动重试(策略)",引用不复述) | 共有 |

### 3.4 默认 ACME 账户

| 配置项 | 业务含义 | 取值口径 | 形态 |
| --- | --- | --- | --- |
| 默认 ACME 账户 | 发起公共 ACME 签发 / 续签时默认使用哪个 ACME 账户(即用哪个 CA、哪个邮箱身份) | 从 acme 已配置账户中选其一;候选清单来源 acme,本模块只存"默认指向";未配置任何账户或默认账户被删除时,签发发起处需引导使用者显式选择账户 / 去 acme 添加(不阻塞 acme 自身账户管理) | 共有 |

## 4. 影响范围:配置项 → 谁消费

> 这是 settings 的核心契约面——每项配置改变会波及哪些模块的业务行为。**引用 certificates 状态机但不复述其流转**。

| 配置项 | 直接消费方 | 消费用途 |
| --- | --- | --- |
| 续签提前天数 | `certificates` | 判定证书是否进入"即将到期"(certificates flow T6;引用不复述)。间接波及:`dashboard` 红点 / 到期高亮(基于"即将到期"状态)、`tasks` 自动续签任务的触发时点 |
| 自动续签开关 | `certificates` | 决定"即将到期"证书是否自动发起续签、以及续签失败项是否在后续扫描自动再尝试;关闭则全部依赖手动。间接波及:`tasks` 是否产生自动续签任务 |
| 默认 ACME 账户 | `certificates` / `acme` | certificates 发起公共 ACME 签发 / 续签时取默认账户;acme 以该账户身份执行验证与签发 |
| 数据存储路径 | `certificates` / `domains` / `acme` / `local-ca` / `tasks` | 各模块的本地持久化数据(证书文件与私钥 / 域名 / ACME 账户密钥 / 根 CA 材料 / 任务历史)均落于此根路径下;属全局基础设施配置 |
| 监听地址 / 监听端口 | 服务器形态自身(接入层) | 决定 Web UI 的网络可达范围与访问边界(project §7 / roles §3);不被其他业务模块消费 |
| 开机自启 | 桌面形态自身(启动行为) | 决定桌面应用是否随系统登录自启常驻;不被其他业务模块消费 |
| 当前运行形态(只读) | 前端形态适配 | 据此决定形态相关设置项(开机自启 vs 监听地址 / 端口)的可见性,并解释各形态特有能力 |

### 决策记录(append-only)

> 只增不改;记"决定了什么 / 为什么不做另一选项"。模块 PRD 的决策记录与此互补(此处记配置项 / 影响范围 / 形态相关决策)。

- **SF1(2026-07-16)· settings 不设实体状态机**:以配置项清单 + 影响范围 + 形态差异刻画,不套状态图。
  - 为什么不做:配置项是"当前取值 + 保存即覆盖",无生命周期流转;强套状态机是伪结构,增理解成本无收益。
- **SF2(2026-07-16)· 续签策略仅含"提前天数 + 自动开关"两项,不设独立重试参数**:不提供"续签重试次数 / 重试间隔"配置。
  - 为什么不做:certificates 的"续签失败自动再尝试"依附自动续签开关的周期扫描(开启即在后续扫描对仍未过期的失败项再发起),独立重试参数需求未明示、徒增配置复杂度;后续确有需要再增量。
- **SF3(2026-07-16)· 不暴露"扫描 / 检测周期"配置**:检测频率为实现机制,不设用户旋钮。
  - 为什么不做:project §7 只承诺"启动即检测 + 定期扫描",周期属实现细节(architect),暴露它属过度配置。
- **SF4(2026-07-16)· 设置内不提供"切换运行形态"**:运行形态是部署既定事实,只读展示。
  - 为什么不做:project §4 / §9-D1 两形态是各自独立部署实例、数据互不共享,运行时切换无意义且会误导使用者以为可迁移数据。
- **SF5(2026-07-16)· 存储路径运行期只读、不可更改(已裁决 orchestrator 2026-07-16)**:路径在部署 / 首次启动时确定,运行期只读展示、不可更改,无迁移语义。
  - 为什么:MVP 最简,避免运行期迁移的复杂度与失败风险;后续确需再增量。
