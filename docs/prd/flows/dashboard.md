## 0. 无独立实体状态机(声明)

总览仪表盘是**聚合视图**,不产生、不拥有、不流转任何核心实体,因此**没有独立状态机**。它对已有实体状态**只读聚合呈现**:

- 证书的当前状态与有效期 ← `certificates`(证书状态机是唯一真相)。
- 签发 / 续签 / 吊销任务的执行结果与时间 ← `tasks`。

本文件因此不画状态流转图,而是定义:**聚合什么、如何聚合、红点 / 高亮由哪些证书状态触发、快速入口去向何处**。

## 1. 模块职责与边界

### 1.1 本模块负责

- **状态总览**:把全部证书按当前状态聚合为概览指标(证书总数 / 即将到期数 / 失败数)。
- **待处理高亮 + 红点**:把处于「需关注 / 需处理」状态的证书汇聚为待处理清单,并以红点提醒使用者「有事要办」。这是 MVP 通知能力的落点(project §5 备注:通知退化为本模块的红点 / 到期高亮)。
- **快速入口**:作为启动首屏,提供跳向常用操作与待处理对象的入口(project §1「启动即检测、看到红点就知道谁该续」)。

### 1.2 委托与消费(边界)

| 事项 | 归属模块 | 本模块的关系 |
| --- | --- | --- |
| 证书当前状态、有效期、状态流转 | `certificates` | **消费**证书状态(证书状态机),据其性质聚合与出红点;本模块不判定、不改变任何证书状态 |
| 「即将到期」的判定(距到期 ≤ 续签策略提前天数) | `certificates`(依 `settings` 策略) | 该判定已由 certificates 依策略写入证书状态(见 flows/certificates.md T6);本模块**只读该既定状态**,不自行计算提前天数、不直接消费 settings 策略 |
| 签发 / 续签 / 吊销任务的执行结果、时间、日志、重试 | `tasks` | **消费**任务执行概况,为待处理项提供「去排查 / 去处理」跳转;任务明细与日志归 tasks,本模块不复述 |
| 域名列表与域名信息 | `domains` | 待处理证书的关联域名**经 certificates 携带**取得,用于显示「是哪个域名的证书该处理」;本模块不直接维护、不直接查询域名 |
| 启动即检测 / 定期扫描以驱动状态流转 | `certificates` | 扫描与状态推进归 certificates(project §7、flows/certificates.md §1.1);本模块的聚合**随证书状态更新而刷新**,不自建扫描机制 |

### 1.3 明确不做

- ❌ 不产生自己的核心实体(无域名 / 证书 / 任务实体,纯聚合)。
- ❌ 不做多渠道通知(邮件 / Webhook / IM);MVP 提醒仅本模块红点 / 高亮(project §6.2)。
- ❌ 不重复实现证书状态判定与扫描(单一出现,归 certificates)。
- ❌ 不主动增设 project §5 未明示的统计维度(如按签发方式 / 按 CA 的分布图表)。

## 2. 聚合逻辑:状态总览三指标

三项概览指标严格对应 project §5,口径**直接映射 certificates §2.1 状态「性质」分类**(引用不复述),据此可判真伪:

| 指标 | 口径(按证书当前状态聚合) | 依据(certificates §2.1「性质」) |
| --- | --- | --- |
| 证书总数 | 当前全部**未删除**证书条目数(含各状态,含终态「已吊销」) | 已删除退出状态机不计;其余状态机内证书全计 |
| 即将到期数 | 处于「即将到期」状态的证书数 | 性质为**稳态·需关注**的状态 |
| 失败数 | 处于「签发失败 / 续签失败 / 已过期」三类状态的证书数之和 | 性质均为**告警·需处理**的状态 |

**口径要点**:

- 「即将到期数」+「失败数」= 全部四类**待处理**证书(即将到期 / 签发失败 / 续签失败 / 已过期),与 §3 红点触发状态完全一致,总览与红点无遗漏、无口径分叉。
- 「即将到期」单列而不并入失败数:其证书**仍有效**,性质是「预警(还能用但快不行了)」;失败数是「已出问题 / 已不可用需处理」,二者语义互补(为什么这样分见 §6-DB1)。
- 「已过期」并入失败数:其性质与签发失败 / 续签失败同为「告警·需处理」;它是「续签 / 签发未成功的到期结果」,归入「问题数」符合业务直觉(为什么这样归见 §6-DB1)。
- 概览数字是快速概括;逐条明细在 §3 待处理清单呈现,两者视角互补,不互相替代。

## 3. 红点 / 待处理高亮触发规则(核心)

红点与待处理高亮由**证书当前状态**触发。触发集 = certificates §2.1 中性质带「需关注」或「需处理」的状态(引用不复述其定义)。

### 3.1 触发状态(出红点 / 进待处理高亮)

| 证书状态(certificates) | 英文标识 | 性质(certificates §2.1) | dashboard 呈现级别 | 使用者应做 |
| --- | --- | --- | --- | --- |
| 已过期 | `expired` | 告警·需处理 | 告警级(最高优先) | 证书已不被信任,尽快续签或吊销删除 |
| 签发失败 | `issue_failed` | 告警·需处理 | 告警级 | 查看失败原因(去 tasks)后重试首签 |
| 续签失败 | `renewal_failed` | 告警·需处理 | 告警级 | 查看失败原因(去 tasks)后重试续签 |
| 即将到期 | `expiring_soon` | 稳态·需关注 | 关注级 | 等自动续签或手动续签;自动策略关闭时尤需关注 |

> 呈现优先级建议(告警 > 关注、已过期居首)是**排序建议**,具体视觉规格(颜色 / 图标 / 排序)在页面 PRD 展开;本表只锚定「哪些状态必须被高亮、各属何级别」这一业务规则。

### 3.2 不触发的状态(说明为何,保持与状态机一致)

| 证书状态 | 性质(certificates §2.1) | 为何不出红点 |
| --- | --- | --- |
| 有效 | 稳态 | 距到期仍长于提前天数,无需处理 |
| 待签发 / 签发中 / 续签中 / 吊销中 | 初始·进行前 / 进行中 | 是**过程态**而非告警;失败后才转入 §3.1 的告警态再出红点 |
| 已吊销 | 终态 | 使用者主动作废,无需提醒;如需重新签发是新动作,不属待处理告警 |

**吊销失败的处理(引用 certificates DC3)**:吊销失败**不设独立告警态**——回退吊销前状态(见 flows/certificates.md T19、DC3),证书本身未变。因此 dashboard **不为「吊销失败」这一事件单独出红点**;若回退后的状态本身属 §3.1 触发集(如回退到「已过期」/「续签失败」),则按该状态正常出红点。此为与证书状态机的一致性约束,dashboard 不新增告警态。

### 3.3 红点聚合、清零与形态差异

- **红点聚合信号**:只要存在**任一**处于 §3.1 触发集的证书,红点即亮;可显示待处理项计数(=「即将到期数」+「失败数」)。红点是「有没有待办」的聚合开关,待处理清单是明细。
- **清零闭环**:当所有待处理项被处置(续签成功→有效、删除、吊销成功→已吊销等)后,触发集为空,红点自动消失——对应 project §1「看到红点就知道谁该续」,处理完即安心。
- **形态差异(引用 glossary「系统托盘」,细节归页面 PRD)**:
  - 桌面形态(仅桌面):红点可同时体现在**系统托盘图标角标**与主窗口内 dashboard;关窗不退出时,托盘角标仍提示待办。
  - 服务器形态:无系统托盘,红点体现在浏览器内 dashboard(可选浏览器标签标题标记)。
  - 上述为运行形态呈现差异,以「仅桌面 / 仅服务器」在页面 PRD 标注;红点的**触发规则(§3.1)两形态一致**。

## 4. 快速入口去向

作为启动首屏,dashboard 提供跳向常用操作与待处理对象的入口。以下为**业务去向**(引用已定页面 / 模块名,不含路由 / API):

| 入口 | 去向 | 目的 |
| --- | --- | --- |
| 待处理项(逐条) | `certificates` 证书详情页(该证书) | 直达出问题 / 需关注的证书,执行续签 / 重试 / 吊销 / 删除 |
| 待处理项的失败原因 | `tasks` 任务历史 / 详情(该证书最近任务) | 查看签发 / 续签为何失败的日志,再决定重试或修正前置 |
| 发起签发 | `certificates` 签发证书页 | 为域名发起首次签发(选域名 + 选签发方式) |
| 查看全部证书 | `certificates` 证书列表页 | 从概览进入完整证书列表 |
| 域名管理 | `domains` 模块 | 维护域名列表(增删改查) |
| 任务与历史 | `tasks` 模块 | 查看全部签发 / 续签 / 吊销执行历史 |

> `certificates` 页面名(证书列表页 / 证书详情页 / 签发证书页)引用自 modules/certificates.md §3;`domains` / `tasks` 页面清单尚未产出,故以模块级去向表述,具体落点页面在各自页面 PRD 定稿后对齐。快速入口的**具体呈现与取舍**在 dashboard 页面 PRD 展开;本表只锚定「首屏应能一键去往哪些业务目的地」。

## 5. 启动即检测与刷新(消费,不自建)

- **启动即检测**(project §7):两形态启动后,由 `certificates` 立即扫描全部证书状态并推进状态流转;dashboard 首屏**读取扫描后的最新状态**,即时呈现总览三指标与红点。dashboard 不自行扫描证书 / 不自行判定到期。
- **持续刷新**:certificates 定期扫描推进状态(如「有效→即将到期」T6、「即将到期→已过期」T10),dashboard 的聚合**随之刷新**,红点与待处理清单反映最新态。刷新的具体机制(实时 / 轮询 / 触发)由 architect 设计,本模块只要求「呈现与证书当前状态一致」。

## 6. 决策记录(append-only)

> 只增不改;记「决定了什么 / 为什么不做另一选项」。模块 PRD 的决策记录与此互补(此处记聚合 / 红点规则相关决策)。

- **DB1(2026-07-16)· 总览口径直接映射证书状态性质分类**:「即将到期数」= 性质「稳态·需关注」;「失败数」= 性质「告警·需处理」(签发失败 + 续签失败 + 已过期)。
  - 为什么这样分:直接锚定 certificates §2.1「性质」列,避免 dashboard 另立一套「什么算失败」的口径造成漂移;二指标合起来恰好覆盖全部四类待处理态,与红点触发集(§3.1)完全一致,无遗漏、无重复口径。
  - 为什么「已过期」并入失败数而非单列:project §5 只列三指标(总数 / 即将到期 / 失败数),已过期性质与签发 / 续签失败同为「告警·需处理」,并入「失败(问题)数」最贴合「有多少证书出问题了」的使用者直觉;单列已过期数会超出 project §5 明示的指标集(避免主动加统计维度)。
- **DB2(2026-07-16)· dashboard 无独立状态机,只读聚合**:不产生核心实体、不判定 / 不改变证书状态、不自建扫描。
  - 为什么:project §5 定位 dashboard 为「状态总览 + 红点 + 快速入口」的聚合视图;证书状态机是全局唯一真相(flows/certificates.md),dashboard 重复判定会制造第二真相源与漂移点。
- **DB3(2026-07-16)· 红点触发集与证书状态机严格一致,不新增告警态**:仅四类状态出红点;吊销失败不出独立红点(引用 certificates DC3,回退后按落地状态判定)。
  - 为什么不为「吊销失败」出红点:certificates DC3 已裁定吊销失败回退原态、不设独立告警态;dashboard 若为其单独出红点,等于凭空引入证书状态机未定义的告警,违反单一出现原则。
- **DB4(2026-07-16)· 通知在 MVP 收敛为本模块红点 / 高亮**:承接 project §5 备注 / §6.2;不做多渠道通知,不单列通知模块。
  - 为什么:MVP 主场景是本机 / 可信内网,启动即见的红点已满足「谁该续」的核心诉求;邮件 / Webhook / IM 价值真实但非首版必需(project §6.2)。
