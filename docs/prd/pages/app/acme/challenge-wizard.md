## 1. 目的

验证方式向导页用于**为域名选择 / 确认验证方式并执行验证挑战**,证明域名控制权。它在**签发 / 续签流程中被触发**(承接 certificates 委托):首签时选择 / 配置验证方式并执行挑战;续签时按已关联方式执行挑战;自动续签遗留的手动挑战也在此承接完成。

本页**对首签与续签不作区分**——两者都是「验证域名 → 取得证书」的同一过程(是否首签由 certificates 的证书状态机区分,本页不感知,DEA1 / flows/acme.md §4.4)。域名↔验证方式的**关联归 domains**,本页提供选择 / 执行体验并持有执行参数(如 HTTP-01 的 webroot 路径,DEA5)。本页**形态无差异**(桌面 / 服务器一致)。

## 2. 功能清单

| # | 功能 | 说明(业务口径) |
| --- | --- | --- |
| F1 验证方式选择 | 为本次签发 / 续签涉及的每个域名选择验证方式:HTTP-01(webroot)或 DNS-01(手动);通配符域名强制 DNS-01;该关联归 domains,本页仅衔接 |
| F2 HTTP-01 配置 | 选 HTTP-01 时指定该域名的 webroot 路径(验证文件放置位置) |
| F3 执行验证挑战 | 为每个域名建立订单、取得挑战、推进挑战状态机(CT1–CT10);HTTP-01 自动放置验证文件并展示自动进度 |
| F4 DNS-01 手动引导 | 展示待添加的 TXT 记录(名 + 值,便于复制)→「等待手动配置」→ 使用者确认已添加 →(可选本地预检)→ 触发校验 |
| F5 挑战失败重试 | 对「验证失败」挑战重试:HTTP-01 重放验证文件 / DNS-01 修正 TXT 后重新展示并重新校验 |
| F6 查看验证 / 挑战状态 | 查看本次各域名挑战当前状态、待处理项与失败原因概要;失败原因日志在任务模块查看 |

> 本页**不含**:DNS 厂商 API 自动验证(MVP 手动,DEA2)、ACME 账户配置(归 ACME 账户页)、证书存储 / 状态 / 到期 / 导出(归 certificates)、批量操作。

## 3. 页面流转

| 触发 | 去向 / 结果 | 说明 |
| --- | --- | --- |
| 选 HTTP-01 并配置 webroot | 系统自动放置验证文件 → 挑战「验证中」→ 通过 /失败 | 校验结束后自动清理验证文件 |
| 选 DNS-01 | 展示待添加 TXT → 挑战「等待手动配置」 | 经总览页红点 / 待处理提醒使用者介入(引用 dashboard) |
| 确认已添加 TXT | (可选本地预检)→ 触发校验 → 挑战「验证中」→ 通过 / 失败 | 通过后提示可移除该 TXT 记录 |
| 挑战失败重试 | 挑战回到「待验证」重新发起 | HTTP-01 重放文件 / DNS-01 重新展示 TXT |
| 全部域名验证通过 | 取得证书并交回 certificates(证书转「有效」)| 本次验证整体成功;证书状态归 certificates |
| 任一域名验证失败 / 已取消 | 交回失败结果 certificates(证书转「签发失败 / 续签失败」)| 可指明哪个域名 |
| 取消手动等待 / 上游任务取消 | 相关挑战转「已取消」 | 中止本次验证,不产生证书 |

> 进入本页的来源包括:签发证书页「验证方式衔接」(首签)、证书详情页发起续签(续签)、总览页待处理项(自动续签遗留的「等待手动配置」挑战)。

## 4. 交互说明

- **触发时机与无差异性**:本页在签发 / 续签流程中被触发(承接 certificates 委托,前提:签发方式为公共 ACME、存在已注册账户);对首签 / 续签执行同一「验证 → 取证」过程,不作区分。
- **验证方式选择**:为每个域名选择 HTTP-01(webroot)或 DNS-01(手动);**通配符域名强制 DNS-01**(glossary);同一次可各域名采用不同方式。该「域名用哪种方式」的关联归 domains,本页仅衔接选择、持有执行参数。
- **HTTP-01(webroot)**:指定该域名 webroot 路径后,系统**自动**在 webroot 放置验证文件并请求 CA 校验,展示自动进度(待验证 → 验证中);全自动、不经「等待手动配置」;校验结束(成功 / 失败均)自动清理验证文件。
- **DNS-01(手动)**:展示待添加的 **TXT 记录**(记录名如 `_acme-challenge.<域名>` 与记录值,内容便于复制),挑战进入「等待手动配置」;使用者在自己的 DNS 服务商处添加该 TXT 后返回**确认已添加**;确认后(可选:提交前本地预检 TXT 是否已生效)请求 CA 校验(转「验证中」);校验通过后提示可移除该 TXT 记录。
- **多域名整体判定**:一次委托可含多个域名(SAN),各域名各跑一个挑战、可分别处于不同状态;**全部**域名挑战「验证通过」本次验证才算成功、方可取证;**任一**域名挑战最终「验证失败 / 已取消」则本次整体失败(可指明哪个域名)(flows/acme.md §3.4)。
- **失败与重试**:「验证失败」挑战可重试——HTTP-01 重放验证文件 / DNS-01 修正 TXT 后重新展示并重新校验;失败原因**日志在任务模块**查看,本页只呈现失败原因概要。
- **等待手动配置的提醒**:DNS-01 停在「等待手动配置」时,经**总览页红点 / 待处理**提醒使用者介入(引用 dashboard,不复述);自动续签在验证环节因手动 DNS-01 停在此态时,使用者经总览页提醒回到本向导完成添加确认后继续。
- **取消**:使用者放弃手动等待,或上游签发 / 续签任务被取消 → 相关挑战转「已取消」,不产生证书。
- **出口交回**:全部通过并取证成功 → 证书交回 certificates(证书转「有效」);任一失败 → 交回失败(证书转「签发失败 / 续签失败」);挑战执行每步时间 / 结果 / 日志经 tasks 留痕(引用不复述)。

## 5. 验收要点

> 业务口径,每条可判真伪。

1. 可为本次签发 / 续签涉及的每个域名选择验证方式:HTTP-01 或 DNS-01(手动)。
2. 通配符域名的验证方式强制为 DNS-01,不可选 HTTP-01。
3. 选 HTTP-01 时可指定该域名的 webroot 路径(验证文件放置位置)。
4. HTTP-01 由系统自动放置验证文件并展示自动进度,不经「等待手动配置」。
5. HTTP-01 校验结束(成功或失败)后自动清理验证文件。
6. 选 DNS-01 时展示待添加的 TXT 记录(记录名与记录值),内容便于复制;挑战进入「等待手动配置」。
7. DNS-01 下使用者确认已添加 TXT 后方可触发校验(挑战转「验证中」)。
8. DNS-01 校验通过后提示可移除该 TXT 记录。
9. 多域名各自跑一个挑战,可分别处于不同挑战状态。
10. 全部域名挑战「验证通过」时本次验证整体成功;任一域名挑战最终「验证失败 / 已取消」则整体失败(可指明哪个域名)。
11. 「验证失败」的挑战可重试:HTTP-01 重放验证文件,DNS-01 修正 TXT 后重新展示并重新校验。
12. 挑战失败原因日志在任务模块查看,本页仅呈现失败原因概要。
13. 「等待手动配置」的待处理提醒由总览页红点 / 待处理呈现,本页不复述。
14. 本向导在签发 / 续签流程中被触发(承接证书模块委托),对首签与续签无差异。
15. 自动续签遇手动 DNS-01 停在「等待手动配置」时,可经本向导完成手动添加确认后继续。
16. 全部验证通过后取得证书并交回证书模块(证书状态归证书模块);任一失败则交回失败结果。
17. 本页形态无差异(桌面 / 服务器一致)。
18. 本页不含 DNS 厂商 API 自动验证、ACME 账户配置(归 ACME 账户页)、证书存储 / 状态管理(归证书模块)。
