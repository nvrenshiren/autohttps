## 1. 角色总览

MVP 只有**一个角色**。两种运行形态的使用者视为**同一信任级别**,不做账户与权限区分。

| 角色   | slug       | 定义                                                              | 认证方式                          |
| ------ | ---------- | ----------------------------------------------------------------- | --------------------------------- |
| 使用者 | `operator` | 桌面形态 = 本机登录用户;服务器形态 = 能访问 Web UI 的可信内网用户 | **无应用层登录**;访问权 = 网络可达性 |

> "访问权 = 网络可达性":谁能连到应用(桌面本机 / 服务器监听范围内),谁就是使用者,拥有全部能力。安全由部署边界(本机 / 内网 / 反向代理)保障,不由应用内鉴权保障。

---

## 2. 权限矩阵(MVP)

单一角色对 7 个模块**全权**(查看 + 变更 + 执行),无细粒度差异。列出各模块关键操作以便验收对齐:

| 模块           | 关键操作                                   | 使用者 |
| -------------- | ------------------------------------------ | :----: |
| `dashboard`    | 查看总览 / 红点 / 待处理项                  |   ✓    |
| `domains`      | 查看 / 新增 / 编辑 / 删除域名               |   ✓    |
| `certificates` | 查看 / 签发 / 续签 / 吊销 / 删除 / 导出证书 |   ✓    |
| `acme`         | 配置账户 / 选择验证方式 / 执行挑战          |   ✓    |
| `local-ca`     | 创建·导入根 CA / 签发内网证书 / 导出根 CA   |   ✓    |
| `tasks`        | 查看历史 / 重试 / 取消任务                  |   ✓    |
| `settings`     | 查看 / 修改全部设置                         |   ✓    |

---

## 3. 访问与信任边界

- **无应用层鉴权**:MVP 不做登录、不区分用户(project §6.2、§9-D4)。
- **依赖部署边界**:
  - 桌面形态:仅本机使用,天然隔离。
  - 服务器形态:默认仅监听本机 / 可信内网(project §7 访问边界)。
- **公网暴露风险**:若将服务器形态暴露到公网,等同于把全部能力开放给任何访问者;需用户自行在前置反向代理加鉴权。产品内应明确提示此风险。

## 4. 后置(不在 MVP)

以下均为 D4 决策下的后续演进项,当前不做:

- 多用户账户体系(注册 / 登录 / 会话)
- 角色分级(管理员 / 操作员 / 只读)
- 每模块 / 每操作细粒度权限
- 操作审计日志(谁在何时做了什么)
